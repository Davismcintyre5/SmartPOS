1. Architecture Overview

SmartPOS follows a three-tier architecture with multiple client applications connected to a central REST API server backed by MongoDB. Desktop and mobile clients include local SQLite databases for offline-first operation.

Data Flow

  1. Online Mode: Client → API Server → MongoDB → Response
  2. Offline Mode (Desktop/Mobile): Client → Local SQLite → Sync Queue → API Server (when online)
  3. Sync Engine: Uploads pending sales, downloads product/customer updates every 30 seconds

Startup Sequence

  1. Load environment variables from .env
  2. Connect to MongoDB with automatic retry (5s interval)
  3. Configure Cloudinary for image uploads
  4. Apply middleware: CORS, JSON parser, Morgan logging, rate limiting
  5. Mount API routes at /api
  6. Start HTTP server on configured port
  7. Register graceful shutdown handlers (SIGTERM, SIGINT)

6. Authentication & Authorization

JWT Token Structure

// Client Token
{
  "id": "user_id",
  "clientId": "business_id",
  "name": "User Name",
  "role": "owner|admin|manager|cashier",
  "type": "client",
  "iat": 1779800000,
  "exp": 1780400000
}

// Admin Token
{
  "id": "admin_id",
  "name": "Admin Name",
  "email": "admin@example.com",
  "role": "admin|superadmin",
  "type": "admin",
  "iat": 1779800000,
  "exp": 1780400000
}

Auth Flow

  1. User logs in with email + password → server validates credentials
  2. Server checks license activation status → returns activated: true/false
  3. If activated → JWT issued with 7-day expiry
  4. If not activated → redirect to activation page (enter license key)
  5. All subsequent requests include Authorization: Bearer <token>
  6. Middleware extracts clientId from JWT for data isolation

Middleware Chain (Client Routes)

Request → auth (verify JWT) → dataIsolation (extract clientId) → licenseCheck (validate license) → Controller

10. AI Integration

SmartPOS integrates with HDM AI (hdmai-server.onrender.com/api/v1) for business intelligence. The AI receives real business data with every request — it never guesses or fabricates information.

Data Gathering Logic

Before sending a request to HDM AI, the aiService.gatherBusinessData() function:

  1. Analyzes the user's message for keywords (sale, revenue, product, stock, customer, report, etc.)
  2. Queries the database for relevant data:
    • Sales data: Last 30 days of completed sales, today's revenue, top products, payment methods
    • Inventory data: All products with stock levels, prices, categories
    • Customer data: All customers with loyalty points, total spent, visits
    • Monthly overview: Current month's sales and transactions
  3. Sends the data along with the user's message to HDM AI
  4. Formats the response (replaces $ amounts with current currency symbol)

11. Payment Integrations

M-Pesa

Supports STK Push, Send Money (manual), Till Number, and Paybill. Configuration managed in admin panel. Callback URL: /api/public/payments/mpesa/callback. Environment toggle between sandbox and production.

Stripe

Card payments via PaymentIntents. Webhook endpoint ready for payment confirmation. Requires STRIPE_SECRET_KEY and STRIPE_WEBHOOK_SECRET.

PayPal

Order creation and capture flow. Requires PAYPAL_CLIENT_ID and PAYPAL_SECRET. Sandbox/production toggle.

Payment Flow

  1. Client selects plan on pricing page → fills registration form → lands on checkout
  2. Selects payment method → follows instructions (M-Pesa manual) or completes payment (Stripe/PayPal)
  3. Clicks "I've Paid" → system creates inactive client + payment record (status: pending)
  4. Admin reviews payment → approves or rejects
  5. On approval: client activated, license generated, user activated, email sent with license key
  6. Unpaid registrations auto-rejected after 3 hours

12. Security Architecture

Authentication

  • JWT tokens signed with HS256, 7-day expiry
  • Passwords hashed with bcrypt (12 salt rounds)
  • Tokens stored in localStorage (web) / AsyncStorage (mobile)
  • Automatic logout on 401 response (interceptor clears storage)

Authorization

  • Role-based access control (Owner > Admin > Manager > Cashier)
  • Granular permissions for each staff role
  • Sidebar items and settings tabs hidden based on permissions
  • Data isolation: every query scoped by clientId from JWT

API Protection

  • CORS whitelist via environment variable
  • Rate limiting: General (200/15min), Auth (20/15min), Payment (10/min)
  • Input validation with Joi schemas on all endpoints
  • Error stack traces hidden in production
  • Maintenance mode returns 503 for all public/client routes

Data Protection

  • TLS 1.3 for all data in transit
  • MongoDB Atlas encryption at rest
  • API keys generated with crypto.randomBytes, displayed masked
  • No credit card numbers or M-Pesa PINs stored
  • Payment processing by PCI-compliant third parties

License Security

  • License keys: SMART-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX format
  • Generated with crypto.randomBytes (not predictable)
  • Device activation required before login
  • Max device limit per license (default: 5)
  • Licenses can be revoked by admin

13. Deployment Guide

Server Deployment

  1. Set up Node.js hosting (Render, DigitalOcean, AWS EC2, or any VPS)
  2. Clone repository: git clone <repo>
  3. Install: cd server && npm install
  4. Configure .env with production values (see Section 3)
  5. Ensure MongoDB Atlas IP whitelist includes server IP
  6. Start: npm start or use PM2: pm2 start server.js --name smartpos
  7. Set up Nginx reverse proxy for HTTPS

Web Client Deployment

  1. Build: cd client && npm run build
  2. Deploy dist/ folder to Vercel, Netlify, or Cloudflare Pages
  3. Set environment variable: VITE_API_BASE_URL=https://smartpos-server.pxxl.click/api

Admin Panel Deployment

  1. Build: cd admin && npm run build
  2. Deploy dist/ to separate subdomain (e.g., hdm-admin.pxxl.click)
  3. Set environment variable: VITE_API_BASE_URL=https://smartpos-server.pxxl.click/api/admin

Desktop App Distribution

  1. Build: cd smartpos-desktop && npm run build
  2. Distribute release/SmartPOS Setup 1.0.0.exe
  3. Configure auto-update server URL in electron/services/updater.js

Mobile App Distribution

  1. Build: cd mobile && eas build --platform android --profile preview
  2. Download APK from Expo dashboard
  3. Distribute via direct download, Google Play Console, or enterprise MDM

14. Maintenance & Backup

Backup System

  • Client Backups: Manual + scheduled. Uses mongodump with clientId filter. Stored in backups/clients/.
  • System Backups: Admin-only. Full database dump or scoped (clients, admin, public). Stored in backups/system/.

Maintenance Mode

Admin can enable maintenance mode from System Settings. All public and client routes return 503. Admin panel remains accessible. Landing page shows maintenance message.

Logging

  • Winston logger: Structured JSON logs
  • Log files: error.log (errors only), combined.log (all levels)
  • Max size: 10MB per file, 5 files retained
  • Console: Colorized output in development

Monitoring Recommendations

  • Set up uptime monitoring (UptimeRobot, Pingdom)
  • Monitor MongoDB Atlas metrics (connections, disk, memory)
  • Watch error logs for unusual patterns
  • Regularly review rate limit hits for potential abuse

15. Troubleshooting

Common Issues

IssueLikely CauseSolution
401 Unauthorized on all requests Token expired or missing Re-login. Check localStorage for token.
403 Forbidden Account suspended or insufficient permissions Contact admin to check account status.
409 Duplicate on customer creation loyaltyCardNumber already exists in another client Fixed in v1.0: compound unique index {clientId, loyaltyCardNumber}
400 Validation failed on sale Missing required fields or invalid values Check request payload against API schema.
MongoDB connection timeout IP not whitelisted in Atlas Add server IP to MongoDB Atlas Network Access.
Desktop app blank screen file:// protocol doesn't support ES modules Use HashRouter for Electron builds.
Mobile build fails (Gradle) newArchEnabled=true with incompatible native modules Set newArchEnabled=false via expo-build-properties plugin.
Receipt showing wrong business name receiptHeader not configured Go to Settings → Receipt and set header.
AI returns "No data provided" data field missing from request Ensure aiService.gatherBusinessData() is called before AI request.
